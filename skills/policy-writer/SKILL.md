# Policy Writer Skill

เขียนนโยบาย/ระเบียบองค์กรภาษาไทย แบบมืออาชีพ ครบถ้วน และนำไปใช้ได้จริง

---

## โครงสร้างมาตรฐาน 11 หัวข้อ

นโยบายทุกชิ้นใช้โครงสร้างนี้เสมอ:

| # | หัวข้อ | เนื้อหา |
|---|--------|---------|
| 1 | วัตถุประสงค์ | ย่อหน้าสั้น อธิบาย "ทำไมถึงมีระเบียบนี้" |
| 2 | ขอบเขตการบังคับใช้ | ระบุว่าใครต้องปฏิบัติตาม ที่ไหน เมื่อไหร่ |
| 3 | สิทธิ์/เงื่อนไขหลัก | ตาราง แสดง tier/ระดับ + หมายเหตุ |
| 4 | สิ่งที่ทำได้ (10 ข้อ) | รายการชัดเจน + ตัวอย่าง เช่น |
| 5 | สิ่งที่ทำไม่ได้ (10 ข้อ) | รายการชัดเจน + วงเล็บทางเลือกที่ถูกต้อง |
| 6 | ขั้นตอนปฏิบัติ | แบ่งย่อย 6.1–6.N แต่ละข้อมี bullet |
| 7 | กรณีพิเศษ/ข้อยกเว้น | 7.1 เหตุการณ์ + 7.2 วิธีปฏิบัติ + หมายเหตุ |
| 8 | ข้อห้ามสำคัญ | 8.1–8.N พร้อมตัวอย่างพฤติกรรมที่ผิด |
| 9 | มาตรการทางวินัย | แบ่งตามประเภทความผิด + บันไดโทษ ครั้งที่ 1/2/3 |
| 10 | ข้อกำหนดเพิ่มเติม | bullet สั้นๆ สิ่งที่ไม่เข้าหมวดอื่น |
| 11 | การบังคับใช้ | วันมีผล + รอบทบทวน + ลายเซ็นปิดท้าย |

### Pattern สำคัญ
- **หัวข้อ 4 & 5** คู่ตรงข้าม (ทำได้ vs ทำไม่ได้) — เสมอ
- **หัวข้อ 6 & 7** คู่ (ปกติ vs ข้อยกเว้น) — เสมอ
- **หัวข้อ 8 & 9** คู่ (ห้าม vs บทลงโทษ) — เสมอ
- ข้อ 4 & 5 ใช้ตัวเลข "10 ข้อ" เพื่อความครบถ้วนและจดจำง่าย
- บทลงโทษหัวข้อ 9 ใช้ "บันไดโทษ" ครั้งที่ 1 → 2 → 3 ขึ้นไป

---

## ขั้นตอนการทำงาน

### Step 1 — เก็บข้อมูลจากผู้ใช้

ถามให้ครบก่อน (ถ้าไม่มีในบริบท):

1. **หัวข้อนโยบาย** คืออะไร? (เช่น การลาป่วย, การแต่งกาย, การใช้รถบริษัท)
2. **ชื่อบริษัท + ที่อยู่** สำหรับใส่ในเอกสาร
3. **วันที่ประกาศใช้**
4. **output ที่ต้องการ**: ในแชท หรือ ไฟล์ Word (.docx)?

ถ้าผู้ใช้ไม่ระบุ ให้ใช้ค่าตัวอย่าง: "บริษัท [ชื่อ] จำกัด" และวันที่ปัจจุบัน

### Step 2 — Draft นโยบาย

เขียนครบทุก 11 หัวข้อ โดย:
- ใช้ภาษาไทยทางการ กระชับ ชัดเจน
- ทุกข้อในหัวข้อ 4, 5 มีตัวอย่างประกอบเสมอ เช่น "เช่น การทำ X, Y"
- ทุกข้อในหัวข้อ 5 มีวงเล็บบอก "ทางเลือกที่ถูกต้อง" เช่น "(ต้องใช้วันลาพักร้อนแทน)"
- บทลงโทษในหัวข้อ 9 ต้องแบ่งตามประเภทความผิด ไม่รวมกัน

### Step 3 — Output

**ถ้าผู้ใช้ต้องการแค่ข้อความ**: แสดงในแชทพร้อม markdown

**ถ้าผู้ใช้ต้องการไฟล์ Word (.docx)**:
→ อ่าน `/mnt/skills/public/docx/SKILL.md` และ `/mnt/skills/user/humanai-docs/SKILL.md`
→ สร้างไฟล์ตาม style มาตรฐาน (ดูหัวข้อ "Word Document Style" ด้านล่าง)

## Word Document Style

เมื่อสร้างไฟล์ .docx ให้ใช้รูปแบบต่อไปนี้:

### Font & สี
- Font: **Niramit** ทุกส่วน
- body: 10pt (size: 20), header/footer: 9pt (size: 18)
- Palette: nearBlack `1A1A1A`, darkGray `2D2D2D`, midGray `595959`, lineGray `D4D4D4`

### Header
```
[ชื่อบริษัทไทย / อังกฤษ] — ชิดซ้าย bold nearBlack
[ที่อยู่] — เว้นบรรทัด ชิดซ้าย midGray
เส้น SINGLE size 6 nearBlack ใต้ header
```

### Footer
```
เส้น SINGLE size 4 lineGray บน footer
[ชื่อระเบียบ  |  วันที่ประกาศ]
```

### Title Block
```
ชื่อระเบียบ — size 52 bold nearBlack (อาจแบ่ง 2 บรรทัด)
ชื่อบริษัท  |  วันที่ — size 18 midGray
rule() เส้นแบ่ง
```

### Section Title
```javascript
// เส้นใต้บางๆ + ข้อความ bold
border: { bottom: { style: BorderStyle.SINGLE, size: 2, color: "D4D4D4" } }
```

### รายการข้อ 4 & 5
- ใช้ **numbered list** (1. 2. 3. ...) ไม่ใช่ bullet
- แต่ละข้อมีตัวอย่าง/วงเล็บ

### ตาราง (หัวข้อ 3)
- Header row: fill `2D2D2D`, text สีขาว bold
- Alternating rows: `FFFFFF` / `F8F8F8`
- Border: SINGLE size 1 color `D4D4D4`

### Page Setup
```
US Letter: width 12240, height 15840
Margin: top/bottom 1080, left/right 1560
Content width: 9120 DXA
```

## ตัวอย่างนโยบายที่รองรับ

| หัวข้อนโยบาย | หัวข้อ 3 (ตาราง) | หัวข้อ 4 & 5 focus |
|---|---|---|
| การลากิจ | สิทธิ์ตามอายุงาน | เหตุผลลาได้ vs ลาไม่ได้ |
| การใช้โทรศัพท์ | ประเภทอุปกรณ์ + สิทธิ์ | กรณีใช้ได้ vs ใช้ไม่ได้ |
| การแต่งกาย | ระดับ formal/smart casual/casual | ชุดที่อนุญาต vs ไม่อนุญาต |
| การใช้รถบริษัท | ประเภทสิทธิ์ตามตำแหน่ง | กรณีใช้ได้ vs ไม่ได้ |
| WFH | วันที่อนุญาต + เงื่อนไข | พฤติกรรมที่อนุญาต vs ไม่อนุญาต |
| การใช้อินเทอร์เน็ต | ประเภทการเข้าถึง | เว็บไซต์ได้ vs ห้าม |
| ค่าล่วงเวลา (OT) | อัตราตามวัน/เวลา | เงื่อนไขขอ OT ได้ vs ไม่ได้ |

## ข้อควรระวัง

- **ห้ามใช้ unicode bullet** "•" หรือ "–" ตรงๆ ใน .docx → ใช้ numbering config เสมอ
- **ห้ามใช้ `\n`** ใน TextRun → แยก Paragraph
- **ตาราง** ต้องมี columnWidths + cell width ทั้งคู่ และต้องรวมได้เท่ากับ TW
- **ShadingType.CLEAR** เสมอ ห้ามใช้ SOLID (จะทำให้พื้นหลังดำ)
- ตรวจสอบด้วย `validate.py` ก่อนส่งให้ผู้ใช้เสมอ
